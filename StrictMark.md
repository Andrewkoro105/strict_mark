Основная идея - разделение структурированого текста и стиля. С возможностью экспорта в word с указанным стилем и с результатами всех расчётов (но с полным определением, гарантируя переносимость файла). И сисиемой расширения похволяющая выпонять блоки разного кода, по надобности объеденяя их в одно пространство. Результатом таких вычислений всегда является либо текст либо картинка.
Система стиля приэтом может не только указывать параметры форматирования блоков, но менять их с помощью lua и настраивать ссылки на элементы втоматически следя за нумерацией.

# Принцип работы программы
При создании проекта создаётся его файл и скрытая папка которая открывается как проект в выбранном редакторе (Рекомендуется VSCode для новичков или любой другой редактор кода для продвинутых пользователей, часть из них можно будет скачать прямо из программы).
Программа будет постоянно пересоздавать docx файл из которого будет генерировать pdf который будет отображён для предпросмотра при запросе.

В docx файл будет вшит проект но без docx c pdf.

Также программа будет поддерживать переносимый файл (либо обновляя его при каждом изменении, либо по зпросу). В нём будут встроенны все файлы проекта, включая кеш от docx с pdf для того чтобы отправив его человеку без моей программы его можно было посмотреть.

Файловая структура выгядит так
- Папака в которуя пользователь положил документ
    - имя_проекта.smz
    - .имя_проекта
        - .git
        - .gitignore
        - style.yaml
        - extensions.yaml
        - main.sm
        иные файлы которые встроит пользователь (напрмер разбивая работу на несколько sm файлов)
        - style_clock
            - style.clock
            иные системые файлы от которых зависит стиль по типу локальных шрифтов, если такие имеются
        - builds
            - cash
                - style
                    скомпидлированные версии lua скриптов
                - sm_cash.cbor (кеши работы всех lua скриптов и резултатов системы расширений)
            - имя_проекта.docx
            - имя_проекта.pdf

В проекте любого пользоватея будет гит который будет создавать коммиты как по запросу так и каждые несколько минут. При создании проекта можно будет указать репозитрий и все коммиты будут улетать туда. `builds` прописанн в `.gitignore`.

# Сили

Стиль напрмую конвертирует моё AST в AST из крейта `docx-rs` или похожего

## Содержимое
У стиял могут быть внешние зависимости, они должны быть прописанны либо ссылкой либо путём к файлу в списке в поле `inputs`

Стиль это yaml файл в котором для каждго типа элементов описанны параметры его форматирования или lua скрипт преобразования для более слоных участков.

Так-же в стиле есть lua скрипты для предварительног редактирования AST (с пмомщью этой системы будут сделаны авто расчёт номеров формул) и его валидации.

Стиль может зависить от другого стиля, как по ссылке так и беря его из вайла системы. В таком случае описания элементов рекурсивно перекрывают другдруга а скрипты редактирования и валидации, а также внешние зависимости применяются все (первыми идут более основные).
В случае указания стиля по ссылке стилем считаетя ссылка должна вести на zip архив со стилем и его lock данными или на git репозиторий (или папку в нём) в котором будет стиль и lock данные. А если стиль на ПК пользователя то он должен былть либо такиже zip архивом либо иметь папку с лок данными рядом.

Под lock данными подрузамевается hash файлов по ссылкам и сами файлы если они  с системы пользователя.

# Структурированный текст
Структура текста описывается модифицированным markdown-ном без поддержки html.

# Структурные элементы
## Имена
Чтобы сослаться на какой либо элемент нужно его проименовать. Это необходимо ибо нумерация принятия в таких случаях может съехать.
Для именования элемента над ним, на отдельной строке, нужно приписать `[elemet name]` а чтобы сослаться нужно указать `@(elemet name)`. Если элемент имеет именованные под элементы то они указываются через `:` например @(elemet name 1):(elemet name 1.1).
У них есть булевый параметр `namespace` Которое превращает его в пространство имён для дочерних объектов, поумолчанию оно false но у списков и таблиц оно true

## Параметры
Некоторые структурные элементы имеют параметры, они указываются с помощью `{}`, и пишутся в формате `name = value`, но будевые значения могут опускать явное описание значения если являются необязательными и их значение поумолчанию false, например: `{bool_param, int_param = 6}`. Параметры пишутся на отдельной строке над плоком к которому пименяются, но ниже иени если оно есть.

## Комментарии
Всё после `%` коментарий и не имеет значение.

## Заголовки
Заголовки помечаются с помощью `#`, чем больше их число тем меньше уровень заголовка.

## Абзацы
Блок текста, для большей читаемости чистого файла в редакторах без авто переноса. Переносы без пустых строк заменяется на пробелы.
Абзацы могут быть разного типа и для его изменения есть параметр `type`, который может принимать значения:
- `text` - обычный абзац текста, значение поумолчанию;
- `footnote` - форматируется как сноска (большинство стилей должно вставлять её в конец страницы на которой она упомянута).

## Цитата
Цитата может содержать любой текст но обрамлённый `<<<` и `>>>`

## Перечисления 
С помощью `-` можно создавать перечисления. Для управления внешним видом есть параметр `{mark_type}`, у него есть несколько значений:
- `number` - в качестве маркеров используются числа, это значение поумолчанию;
- `mark` - в качестве маркеров используется константный маркер ввбраный стилем;
- `char` - в качестве маркеров используются буквы;
- `bibliography` - весь список и ссылки на него форматируется как библиографический список;
- `definitions` - весь список и ссылки на него форматируется как список определений.
Чтобы сослаться на конкретный элемент можно использовать его номер или имя если указать (второй вариант более рекомендуем). Текст внутри выравнивания с помощью пробелов к своему началу и пишется как обычный текст.
Если текст является сноской то можно установить параметр `type` в значение `footnote`


## Форматирование
Текст можно сделать жирным `**жирный текст**`, курсивным `*курсивный текст*` , подчёркнутым `__подчёркнутый текст__`,  зачёркнутым `~~зачёркнутый текст~~` и неделимый текст с помощью \`.

## Формулы
В одинарных `$` можно будет писать внутристрочные формулы на LaTeX, а в двойных те что будут вынесены из текста и пронумерованны многими стилями.

## Иные файлы
Иные файлы вставляются похожим образом но делятся на 2 типа, вставка страницей и вставка ресурсом.

### Вставка страницей
Вставляет изображения и pdf как страницы `(path)`

### Вставка ресурсом
Вставляет изображения добавляя к ним подписи, если несколько таких изображений указаны подряд без пустых строк, то они будут объединены в группу.
```
(path1.png)"
    text1
"
(path2.png)"
    text2
"
```

### Вставка sm файла
Вставляет указанный файл вставляя список его данных в место вызыва вставки `(path.sm)`

## Таблицы
Таблицы можно создавать похожим на markdown способо, но он расширен дабы дать возможность объединять ячейки и указывать как в ячейках должен распологатся текст.

К сожалению в документах таблица нужно оченб много функций но обычно они не используются все сзразу так что оно не должно выглядеть супер сложно.

Между каждой строкой идёт разделение с чётким указанием сколько ячеек в следующей и как в них расположены данные. Всё такие разделители должны чётко совпадать по числу символов а разделители колонок должны стоять в одном столбце чтобы указывать край одной колонки. Это единстаенно место с подобномным требованием во всех остальных случаях это не обязательно, но будет исправленно форматером.
Колонки разделяются символами указвающими как в ячейке снизу справа от разделителя располагается текст. Доступны 3 варианта:

- `|` - поцентру
- `^` - сверху
- `v` - снизу

Около разделителей могут стоять `:` они указывают к какому краю равнять текст в ячейке подними, если их нет то текст располагается по середине (ставить двоеточия на оба края запрещено).
Оставшаяся место между разделителями должно быть заполнено `-`. Если строка сверху это шапка то после строки должен стаять знак `<<`.
 
Если нужно объединить ячейки по вертикали то в качесве левого разделителя колонки нужно использовать только `|` и сделав отступ из пробелов внутрб яцейки писать текст. 
При этом в ткой части разделителя не важно число символов, но остальная чать должна ставится так будто в это все стоит верно. Такой подход нужен чтобы в уже созданной структуре можно было легко педактровать текст, использовать его для ленивого описания структуры зачастую будет неудоно.

Строки разделяются ` | ` число колонок должно совпадать в разметкой данной строки, но положение самих разделителей может не совпадать, это будет исправлять форматер. Текст внутри может быть многострочным, к нему применяются те-же системы что и ко всему файлу.

Также любой колонке можно дать имя и тип чтобы к ней можно было обращаться из расширений. Для этого нужно в добавить к разделю строк который форматирует верхную ячейку новую строку в которой левый разделитель начальной ячейки будет `|` и через `-` будет указанны `[name]type`, оставшееся простанство до следующего разделителя следует заполнить `-`, число столбцов должно совпадать с разделителем строк выше, но разделители колонок не обязанны, это поправит форматр. Приэтом все зраделители этой колонки должны состоять тольоко из `|` ибо форматирование ячеек наследуюется из первой. А сами ячейки теперь не требуют построчных разделителей ибо их данные занимаю 1 строку (пустые строоки игнорируются). Но важно, хоть и рекомендуется использовать эту систему не для всех колонок в рамках одних и техже строк, это вохможно но требует строгово соблюдения числа строк. Также можно создавать не именнованные данные, в таком случае первый постоянный сивол `-` в разделителе над первой ячейкой нужно заменить на 'd'

У таблицы есть параметр `size` в котром указан список размеров колонок (сумма элементов в неём должна быть равна 1, а их число должно совпаать с числом клонок)

После таблицы в кавычках идёт описание таблицы, тест внутри имеет обязательный отступ если он пногострочный.

В примере ниже первая буква указывает на то как будет выравнен текст по вертикали, вторая по горизонтали. В ячейках на которые ссылается пояснение стоят их номера
```
[table name]
{size = [50%, 20%, 30%]}
^:-------------|------v-----:|
| вл           | цц   | нп 1 |
|---------------------| нп 1 |
| цц 2                | нп 1 |
|-------------:^-------------|
| цп           | вц          |
|-d-----------:|-------------|<<
|-[name]type---|-------------|
| цп 3         | цц 5        |
|              |:------------|
| цп 4         | цл 6        |
v-------------:|             |
| нп           |             |
"
    Таблица с самым полным примером синтаксиса (в реальности таких небудет ибо это обсурдное нагромождение всех функций).

    Сначало указанно имя таблицы, потом соотношение размеров через параметр.
    Ячейка 1 объеденена по вертикали так как разделитель строк под ней состоит не из `-`, её текст это: "нп 1\nнп 1\nнп 1"
    Ячейка 2 объеденена по горизонтали так как разделитель строк над ней не содержит 1 разделителя колонок.
    Ячейки 3 и 4 это 2 ячейки именнованного набора. Их должно быть 2 так как сбоку от них 2 ячейки 5 и 6, не смотря на то что ячейка 6 объеденена по вертикали с ячейкой ниже
"

[table name 2]
|---|---------------|
| Имена | этапы ихмерений |
|   |---|---|---|---|
|   | 1 | 2 | 3 | 4 |
|-d:|-d-|-d-|-d-|-d-|<<
| Anedew | 30 | 40 | 20 | 50 |
| Bob | 30 | 40 | 20 | 50 |
| Alex | 30 | 40 | 20 | 50 |
| Jon | 30 | 40 | 20 | 50 |"
    То как напишет таблицу человек, до обработки форматером
"

[table name 2 (result)]
|--------|-----------------------|
| Имена  | этапы измерений       |
|        |-----|-----|-----|-----|
|        | 1   | 2   | 3   | 4   |
|-d-----:|-d---|-d---|-d---|-d---|<<
| Anedew | 30  | 40  | 20  | 50  |
| Bob    | 30  | 40  | 20  | 50  |
| Alex   | 30  | 40  | 20  | 50  |
| Jon    | 30  | 40  | 20  | 50  |"
    То что выдаст форматер
"
```

## Блоки кода
Блоки кода это основной способ расширения возможностей редактора.
В специальном файле можно прописать какая программа исполнит этот код и как извлеч её результат. Но в конце концов результатом всегда будет текст в формате этого редактора и набор ресурсов (по типу изображений) которые этот текстобязан вставить.

Блоки кода прописываются как и в markdown но вместо языка указывается название расширения в котором настроено исполнение этого кода.

Например так будут реализованны диаграммы, авто расчёт математики, LaTeX.
