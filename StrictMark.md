Основная идея - разделение структурированного текста и стиля. С возможностью экспорта в Word с указанным стилем и результатами всех расчётов, при этом обеспечивая максимальную переносимость файла. И системой расширения, позволяющей выполнять блоки разного кода, по необходимости объединяя их в единое пространство. Результатом таких вычислений всегда является либо текст, либо файлы на которые этот текст должен ссылаться.
Система стиля при этом может не только указывать параметры форматирования блоков, но и изменять их с помощью Lua, а также настраивать ссылки на элементы автоматически, следя за нумерацией.

# Принцип работы программы
При создании проекта создаётся его файл и скрытая папка, которая открывается как проект в выбранном редакторе (рекомендуется VS Code для новичков или любой другой редактор кода для продвинутых пользователей; некоторые из них можно будет скачать прямо из программы).
Программа будет постоянно пересоздавать DOCX-файл, из которого будет генерировать PDF. PDF будет отображаться для предпросмотра по запросу.

В DOCX-файл будет встроен проект, но без самого DOCX и PDF (только данные).

Также программа будет поддерживать переносимый формат файла (кнопки сохранения не будет - сохранения всегда автоматические). В нём будут встроены все файлы проекта, включая кеш от DOCX и PDF, чтобы, отправив его человеку без моей программы, его можно было просмотреть.
Также в нём все SM-файлы будут отформатированы, а текст во всех абзацах перенесён так, чтобы длина строк не превышала 200 символов; при необходимости будет проще читать и менять SM-файлы вне специальных редакторов. Все новые переносы будут помечены специальным символом.

Файловая структура выглядит так:
- Папка, в которую пользователь положил документ
    - имя_проекта.smz
    - .имя_проекта
        - .git
        - .gitignore
        - style.yaml
        - extensions.yaml
        - main.sm
        иные файлы, которые встроит пользователь (например, разбивая работу на несколько SM-файлов)
        - style.clock
        - style_clock.zip
            иные системные файлы, от которых зависит стиль (например, локальные шрифты), если такие имеются
        - builds
            - имя_проекта.docx
            - имя_проекта.pdf
            - cache
                - style
                    скомпилированные версии Lua-скриптов
                - sm_cache.cbor (кеш работы всех Lua-скриптов и результатов системы расширений)

В проекте любого пользователя будет Git, который будет создавать коммиты как по запросу, так и каждые несколько минут, а также при любом сохранении. При создании проекта можно будет указать репозиторий, и все коммиты будут улетать туда. Более того, для популярных Git-хранилищ будут реализованы системы автоматической настройки доступа и создания репозитория, которые можно будет расширить в настройках и на другие платформы. `builds` прописан в `.gitignore`.

# Стиль
Стиль напрямую конвертирует моё AST в AST из крейта `docx-rs` или аналогичного.

## Содержимое
У стиля могут быть внешние зависимости; они должны быть прописаны либо ссылкой, либо путём к файлу в списке в поле `inputs`.

Стиль - это YAML-файл, в котором для каждого типа элементов описаны параметры его форматирования или Lua-скрипт преобразования для более сложных участков.

Также в стиле есть Lua-скрипты для предварительного редактирования AST (с помощью этой системы будут сделаны авторасчёт номеров формул) и его валидации.

Стиль может зависеть от другого стиля как по ссылке, так и беря его из файла системы. В таком случае описания элементов рекурсивно переопределяют друг друга, а скрипты редактирования и валидации, а также внешние зависимости применяются все вместе (первыми идут более основные).
В случае указания стиля по ссылке, ссылка должна вести на ZIP-архив со стилем и его lock-данными или на Git-репозиторий (или папку в нём), в котором будет стиль и lock-данные. Если стиль находится на ПК пользователя, то он должен быть либо таким же ZIP-архивом, либо иметь папку с lock-данными рядом.

Под lock-данными подразумевается хэш файлов по ссылкам и сами файлы, если они с системы пользователя.

# Структурированный текст
Структура текста описывается модифицированным Markdown-ом без поддержки HTML.

# Структурные элементы
## Имена
Чтобы сослаться на какой-либо элемент, нужно его переименовать. Это необходимо, ибо нумерация в таких случаях может съехать.
Для именования элемента над ним, на отдельной строке, нужно приписать `[element name]`, а чтобы сослаться - указать `@(element name)`. Если элемент имеет именованные подэлементы, то они указываются через `:`, например `@(element name 1):(element name 1.1)`.
У них есть булевый параметр `namespace`, который превращает его в пространство имён для дочерних объектов; по умолчанию он false, но у списков и таблиц он true.

## Параметры
Некоторые структурные элементы имеют параметры; они указываются с помощью `{}` и пишутся в формате `name = value`, но булевы значения могут опускать явное описание значения, если являются необязательными и их значение по умолчанию false, например: `{bool_param, int_param = 6}`. Параметры пишутся на отдельной строке над блоком, к которому применяются, но ниже имени, если оно есть.

## Комментарии
Всё после `%` - комментарий и не имеет значения.

## Заголовки
Заголовки помечаются с помощью `#`; чем больше их число, тем меньше уровень заголовка.

## Абзацы
Блок текста; для большей читаемости чистого файла в редакторах без авто переноса, переносы без пустых строк заменяются на пробелы.
Абзацы могут быть разного типа, и для их изменения есть параметр `type`, который может принимать значения:
- `text` - обычный абзац текста, значение по умолчанию;
- `footnote` - форматируется как сноска (большинство стилей должно вставлять её в конец страницы, на которой она упомянута).

## Цитата
Цитата может содержать любой текст, обрамлённый `<<<` и `>>>`.

## Перечисления
С помощью `-` можно создавать перечисления. Для управления внешним видом есть параметр `{mark_type}`, у него есть несколько значений:
- `number` - в качестве маркеров используются числа, это значение по умолчанию;
- `mark` - в качестве маркеров используется константный маркер, выбранный стилем;
- `char` - в качестве маркеров используются буквы;
- `bibliography` - весь список и ссылки на него форматируется как библиографический список;
- `definitions` - весь список и ссылки на него форматируется как список определений.
Чтобы сослаться на конкретный элемент, можно использовать его номер или имя, если указано (второй вариант более предпочтителен). Текст внутри выравнивается с помощью пробелов по началу строки и пишется как обычный текст.
Если текст является сноской, то можно установить параметр `type` в значение `footnote`.

В списках каждый элемент является полноценным SM-текстом со всеми возможностями. При написании более одной строки можно использовать для каждой последующей либо табуляцию, либо 2 пробела. Так как некоторые текстовые редакторы убирают пробельные символы на пустых строках, то рекомендуется для нескольких абзацев оборачивать текст в `{}`; в таком режиме отступ лишь рекомендация.

## Форматирование
Текст можно сделать жирным `**жирный текст**`, курсивным `*курсивный текст*`, подчёркнутым `__подчёркнутый текст__`, зачёркнутым `~~зачёркнутый текст~~` и кодом (моноширинным) с помощью обратных кавычек.

## Формулы
В одинарных `$` можно будет писать внутристрочные формулы на LaTeX, а в двойных - те, что будут вынесены из текста и пронумерованы многими стилями.

## Иные файлы
Вставка файлов в документ делится на 3 типа: вставка страницей, вставка ресурсом и вставка SM или SMZ-файла.

Если файл не находится в папках проекта, то он копируется как кеш.

### Вставка страницей
Вставляет изображения и PDF как страницы `(path)`.

### Вставка ресурсом
Вставляет изображения, добавляя к ним подписи. Если несколько таких изображений указаны подряд без пустых строк, то они будут объединены в группу.
```
(path1.png)"
text1
"
(path2.png)"
text2
"
```

### Вставка SM или SMZ-файла
Вставляет указанный файл, вставляя список его данных в место вызова `(path.sm)`.

## Таблицы
Таблицы можно создавать похожим на Markdown способом, но он расширен, чтобы дать возможность объединять ячейки и указывать, как в ячейках должен располагаться текст.

К сожалению, в документах таблицам нужно очень много функций, но обычно они не используются все сразу, так что это не должно выглядеть слишком сложно.

Между каждой строкой идёт разделение с чётким указанием, сколько ячеек в следующей и как в них расположены данные. Все такие разделители должны чётко совпадать по числу символов, а разделители колонок должны стоять в одном столбце, чтобы указывать границы колонок. Это единственное место с подобным требованием; во всех остальных случаях это не обязательно, но будет исправлено форматером.
Колонки разделяются символами, указывающими, как в ячейке снизу справа от разделителя располагается текст. Доступны 3 варианта:

- `|` - по центру
- `^` - сверху
- `v` - снизу

Около разделителей могут стоять `:`, они указывают, к какому краю выравнивать текст в ячейке под ними; если их нет, то текст располагается посередине (ставить двоеточия на оба края запрещено).
Оставшееся место между разделителями должно быть заполнено `-`. Если строка сверху - это шапка, то после строки должен стоять знак `<<`.
 
Если нужно объединить ячейки по вертикали, то в качестве левого разделителя колонки нужно использовать только `|` и, сделав отступ из пробелов внутри ячейки, писать текст.
При этом в такой части разделителя не важно число символов, но остальная часть должна ставиться так, будто в этой части всё стоит верно. Такой подход нужен для того, чтобы в уже созданной структуре можно было легко редактировать текст; использовать его для ленивого описания структуры зачастую будет неудобно.

Строки разделяются `|`, при этом число колонок должно совпадать с разметкой данной строки, но положение самих разделителей может не совпадать - это будет исправлять форматер. Текст внутри может быть многострочным, к нему применяются те же системы, что и ко всему файлу.

Также любой колонке можно дать имя и тип, чтобы к ней можно было обращаться из расширений. Для этого нужно добавить к разделителю строк, который форматирует верхнюю ячейку, новую строку, в которой левый разделитель начальной ячейки будет `|`, и через `-` будут указаны `[name]type`; оставшееся пространство до следующего разделителя следует заполнить `-`; число столбцов должно совпадать с разделителем строк выше, но разделители колонок не обязаны - это поправит форматер. При этом все разделители этой колонки должны состоять только из `|`, ибо форматирование ячеек наследуется из первой. А сами ячейки теперь не требуют построчных разделителей, ибо их данные занимают 1 строку (пустые строки игнорируются). Но важно: хотя рекомендуется использовать эту систему не для всех колонок в рамках одних и тех же строк, это возможно, но требует строгого соблюдения числа строк. Также можно создавать неименованные данные; в таком случае первый постоянный символ `-` в разделителе над первой ячейкой нужно заменить на `'d'`.

У таблицы есть параметр `size`, в котором указан список размеров колонок (сумма элементов в нём должна быть равна 1, а их число должно совпадать с числом колонок).

После таблицы в кавычках идёт описание таблицы; текст внутри имеет обязательный отступ, если он многострочный.

В примере ниже первая буква указывает на то, как будет выровнен текст по вертикали, вторая - по горизонтали. В ячейках, на которые ссылается пояснение, стоят их номера.
```
[table name]
{size = [50%, 20%, 30%]}
^:-------------|------v-----:|
| вл           | цц   | нп 1 |
|---------------------| нп 1 |
| цц 2                | нп 1 |
|-------------:^-------------|
| цп           | вц          |
|-d-----------:|-------------|<<
|-[name]type---|-------------|
| цп 3         | цц 5        |
|              |:------------|
| цп 4         | цл 6        |
v-------------:|             |
| нп           |             |
"
    Таблица с самым полным примером синтаксиса (в реальности таких не будет, ибо это абсурдное нагромождение всех функций).

    Сначала указано имя таблицы, потом соотношение размеров через параметр.
    Ячейка 1 объединена по вертикали, так как разделитель строк под ней состоит не из `-`, её текст: "нп 1\nнп 1\nнп 1".
    Ячейка 2 объединена по горизонтали, так как разделитель строк над ней не содержит 1 разделителя колонок.
    Ячейки 3 и 4 - это 2 ячейки именованного набора. Их должно быть 2, так как сбоку от них 2 ячейки 5 и 6, несмотря на то что ячейка 6 объединена по вертикали с ячейкой ниже.
"

[table name 2]
|---|---------------|
| Имена | этапы измерений |
|   |---|---|---|---|
|   | 1 | 2 | 3 | 4 |
|-d:|-d-|-d-|-d-|-d-|<<
| Anedew | 30 | 40 | 20 | 50 |
| Bob | 30 | 40 | 20 | 50 |
| Alex | 30 | 40 | 20 | 50 |
| Jon | 30 | 40 | 20 | 50 |"
    То, как напишет таблицу человек до обработки форматером.
"

[table name 2 (result)]
|--------|---------------------------------------|
| Имена  | этапы измерений                       |
|        |---------|---------|---------|---------|
|        | 1       | 2       | 3       | 4       |
|-d-----:|-d-------|-d-------|-d-------|-d-------|<<
| Anedew | 30      | 40      | 20      | 50      |
| Bob    | 30      | 40      | 20      | 50      |
| Alex   | 30      | 40      | 20      | 50      |
| Jon    | 30      | 40      | 20      | 50      |"
    То, что выдаст форматер.
"
```

## Блоки кода
Блоки кода - это основной способ расширения возможностей редактора.
В специальном файле можно прописать Lua-скрипт, который будет либо выполнять работу сам, либо вызывать стороннее ПО для обработки (вплоть до компиляции кода других языков). Но в конечном счёте результатом всегда будет текст в формате этого редактора и набор ресурсов (например, изображения), которые этот текст обязан вставить.

Блоки кода прописываются, как и в Markdown, но вместо языка указывается название расширения, в котором настроено исполнение этого кода.

Например, так будут реализованы диаграммы, авторасчёт математики, Typst и LaTeX.

Также эти расширения имеют доступ ко всем переменным файла.
Если расширение опирается на переменные, которые ещё не сгенерированы, и при этом не может вывести их из своего кода, то их нужно указать явно. В таком случае выполнение расширения будет отложено, пока в контексте не появятся новые переменные.

Часто необходимо объединять расширения в цепочки, чтобы вся информация между ними передавалась нативно. Для этого у нужных расширений нужно указать имя цепочки с помощью `@`.

```
    ```@chain_name exception_name (@(имя переменной 1), @(имя переменной 2))
        code
    ```
```

После исполнения расширение должно сказать, какие переменные оно использовало; это необходимо для оптимизации пересчётов.

Расширения могут полагаться на внешние зависимости, и в таком случае они должны либо сами устанавливать и настраивать их, либо давать пользователю инструкцию. А перед началом работы они должны собрать всю информацию для воспроизводимости и записать её в lock-файл.

Некоторые расширения могут частично компилировать свой код для ускорения работы.

Все блоки расширений могут собраться в C-подобную библиотеку. Если расширение не предоставляет такой функционал, его пайплайн вызова собирается в Zig, а его зависимости прописываются в специальном YAML-файле зависимостей этой библиотеки, с пайплайном воспроизведения окружения и lock-файлом, если он есть.

# Вызов `sm` и `smz` файлов как программ
`sm` и `smz` можно собрать в программу и исполнить. Для этого нужно указать, какие переменные будут меняться и какую переменную нужно вычислить. Тогда все необходимые блоки расширений будут собраны в C-подобные библиотеки, а после сгенерирован и собран Zig-код, который их объединяет в единую программу, а также проверяет окружение и воспроизводит необходимое, если это возможно (если нет - выдаёт ошибку с описанием, как наладить окружение).

```sh
sm main.sm build fn_name(@(имя переменной 1), @(имя переменной 2)) -> @(имя переменной 3)
```

```sh
sm main.sm run fn_name(@(имя переменной 1) = 3, @(имя переменной 2) = 4) -> @(имя переменной 3)
```

# Алгоритм сборки
0. Сборка стиля (загрузка всех зависимостей, сборка Lua-скриптов)
1. Парсинг
2. Выполнение расширений
    1. Сборка всех расширений, которым это нужно (с кешированием)
    2. Вычисление необходимых входных переменных для каждого блока
    3. Все блоки начинают собираться по возможности параллельно и асинхронно; после сборки блок вставляется в текст, и вычисляется, какие переменные он добавил; если с их добавлением часть блоков становятся вычислимыми, они вычисляются. Результат выполнения всех блоков кешируется (с записью всех его зависимостей).
3. Валидация документа.
4. Редактирование документа стилем.
5. Расчёт форматирования каждого блока с кешированием (конвертация в DOCX).
