Основная идея - разделение структурированного текста и стиля. С возможностью экспорта в word с указанным стилем и с результатами всех расчётов (но с полным определением, гарантируя переносимость файла). И системой расширения позволяющая выполнять блоки разного кода, по надобности объединяя их в одно пространство. Результатом таких вычислений всегда является либо текст, либо картинка.
Система стиля при этом может не только указывать параметры форматирования блоков, но менять их с помощью lua и настраивать ссылки на элементы автоматически следя за нумерацией.

# Принцип работы программы
При создании проекта создаётся его файл и скрытая папка которая открывается как проект в выбранном редакторе (Рекомендуется VSCode для новичков или любой другой редактор кода для продвинутых пользователей, часть из них можно будет скачать прямо из программы).
Программа будет постоянно пересоздавать docx файл из которого будет генерировать pdf. Который будет отображён для предпросмотра при запросе.

В docx файл будет вшит проект, но без docx c pdf.

Также программа будет поддерживать переносимый файл (кнопки сохранить не будет, сохранения всегда автоматические). В нём будут встроены все файлы проекта, включая кеш от docx с pdf для того чтобы отправив его человеку без моей программы его можно было посмотреть. Также в нём все sm файлы будут отфарматированны а текст во всех обзацах перенесён так чтоб длина строк не привыхала 200 символов, так прии необходимости будет проще читать и менять sm файле вне специальных редакторов. Все новые переносы будут помечены специальным символом

Файловая структура выглядит так
- Пака в которую пользователь положил документ
    - имя_проекта.smz
    - .имя_проекта
        - .git
        - .gitignore
        - style.yaml
        - extensions.yaml
        - main.sm
        иные файлы которые встроит пользователь (например разбивая работу на несколько sm файлов)
        - style.clock
        - style_clock.zip
            иные системные файлы от которых зависит стиль по типу локальных шрифтов, если такие имеются
        - builds
            - имя_проекта.docx
            - имя_проекта.pdf
            - cash
                - style
                    скомпилированные версии lua скриптов
                - sm_cash.cbor (кеши работы всех lua скриптов и результатов системы расширений)

В проекте любого пользователя будет git который будет создавать коммиты как по запросу, так и каждые несколько минут, так и при любом сохранении. При создании проекта можно будет указать репозиторий и все коммиты будут улетать туда. Более того для популярных git хранилиь будут реализоанны системы автоматической настройки поступа и создания репозитория, которые можно будет расширить в настройках и на други платформы. `builds` прописан в `.gitignore`.

# Сили

Стиль напрямую конвертирует моё AST в AST из крейта `docx-rs` или похожего

## Содержимое
У стиля могут быть внешние зависимости, они должны быть прописаны либо ссылкой, либо путём к файлу в списке в поле inputs

Стиль это yaml файл в котором для каждого типа элементов описаны параметры его форматирования или lua скрипт преобразования для более сложных участков.

Также в стиле есть lua скрипты для предварительного редактирования AST (с помощью этой системы будут сделаны авто расчёт номеров формул) и его валидации.

Стиль может зависеть от другого стиля, как по ссылке, так и беря его из файла системы. В таком случае описания элементов рекурсивно перекрывают друг друга, а скрипты редактирования и валидации, а также внешние зависимости применяются все (первыми идут более основные).
В случае указания стиля по ссылке стилем считается ссылка должна вести на zip архив со стилем и его lock данными или на git репозиторий (или папку в нём) в котором будет стиль и lock данные. А если стиль на ПК пользователя то он должен быть либо таким же zip архивом либо иметь папку с lock данными рядом.

Под lock данными подразумевается hash файлов по ссылкам и сами файлы если они с системы пользователя.

# Структурированный текст
Структура текста описывается модифицированным markdown-ном без поддержки html.

# Структурные элементы
## Имена
Чтобы сослаться на какой-либо элемент, нужно его переименовать. Это необходимо ибо нумерация принятия в таких случаях может съехать.
Для именования элемента над ним, на отдельной строке, нужно приписать `[elemet name]` а чтобы сослаться, нужно указать `@(elemet name)`. Если элемент имеет именованные под элементы, то они указываются через `:` например `@(elemet name 1):(elemet name 1.1)`.
У них есть булевый параметр  namespace  Которое превращает его в пространство имён для дочерних объектов, по умолчанию оно false, но у списков и таблиц оно true

## Параметры
Некоторые структурные элементы имеют параметры, они указываются с помощью `{}`, и пишутся в формате `name = value`, но будевые значения могут опускать явное описание значения если являются необязательными и их значение по умолчанию false, например: `{bool_param, int_param = 6}`. Параметры пишутся на отдельной строке над блоком к которому поменяются, но ниже имени если оно есть.

## Комментарии
Всё после `%` комментарий и не имеет значение.

## Заголовки
Заголовки помечаются с помощью `#`, чем больше их число тем меньше уровень заголовка.

## Абзацы
Блок текста, для большей читаемости чистого файла в редакторах без авто переноса. Переносы без пустых строк заменяется на пробелы.
Абзацы могут быть разного типа и для его изменения есть параметр  type , который может принимать значения:
- `text` - обычный абзац текста, значение по умолчанию;
- `footnote` - форматируется как сноска (большинство стилей должно вставлять её в конец страницы на которой она упомянута).

## Цитата
Цитата может содержать любой текст, но обрамлённый `<<<` и `>>>`

## Перечисления
С помощью `-` можно создавать перечисления. Для управления внешним видом есть параметр `{mark_type}`, у него есть несколько значений:
- `number` - в качестве маркеров используются числа, это значение по умолчанию;
- `mark` - в качестве маркеров используется константный маркер выбранный стилем;
- `char` - в качестве маркеров используются буквы;
- `bibliography` - весь список и ссылки на него форматируется как библиографический список;
- `definitions` - весь список и ссылки на него форматируется как список определений.
Чтобы сослаться на конкретный элемент можно использовать его номер или имя если указать (второй вариант более рекомендуем). Текст внутри выравнивания с помощью пробелов к своему началу и пишется как обычный текст.
Если текст является сноской, то можно установить параметр `type` в значение `footnote`


## Форматирование
Текст можно сделать жирным `**жирный текст**`, курсивным `*курсивный текст*`, подчёркнутым `__подчёркнутый текст__`, зачёркнутым `~~зачёркнутый текст~~` и неделимый текст с помощью `\``.

## Формулы
В одинарных `$` можно будет писать внутри строчные формулы на LaTeX, а в двойных те что будут вынесены из текста и пронумерованы многими стилями.

## Иные файлы
Иные файлы вставляются похожим образом, но делятся на 2 типа, вставка страницей и вставка ресурсом.

### Вставка страницей
Вставляет изображения и pdf как страницы `(path)`

### Вставка ресурсом
Вставляет изображения добавляя к ним подписи, если несколько таких изображений указаны подряд без пустых строк, то они будут объединены в группу.
```
(path1.png)"
text1
"
(path2.png)"
text2
"
```

### Вставка sm файла
Вставляет указанный файл вставляя список его данных в место вызова вставки `(path.sm)`

## Таблицы
Таблицы можно создавать похожим на markdown способом, но он расширен дабы дать возможность объединять ячейки и указывать как в ячейках должен располагаться текст.

К сожалению в документах таблицам нужно очень много функций, но обычно они не используются все сразу, так что оно не должно выглядеть супер сложно.

Между каждой строкой идёт разделение с чётким указанием сколько ячеек в следующей и как в них расположены данные. Всё такие разделители должны чётко совпадать по числу символов, а разделители колонок должны стоять в одном столбце, чтобы указывать край одной колонки. Это единственно место с подобным требованием во всех остальных случаях это не обязательно, но будет исправлено форматером.
Колонки разделяются символами указывающими как в ячейке снизу справа от разделителя располагается текст. Доступны 3 варианта:

- `|` - по центру
- `^` - сверху
- `v` - снизу

Около разделителей могут стоять `:` они указывают к какому краю равнять текст в ячейке подними, если их нет то текст располагается посередине (ставить двоеточия на оба края запрещено).
Оставшаяся место между разделителями должно быть заполнено `-`. Если строка сверху это шапка, то после строки должен стаять знак `<<`.
 
Если нужно объединить ячейки по вертикали, то в качестве левого разделителя колонки нужно использовать только `|` и сделав отступ из пробелов внутри ячейки писать текст.
При этом в такой части разделителя не важно число символов, но остальная часть должна ставится так будто в этой все стоит верно. Такой подход нужен чтобы в уже созданной структуре можно было легко редактировать текст, использовать его для ленивого описания структуры зачастую будет неудобно.

Строки разделяются `|`, при этом число колонок должно совпадать с разметкой данной строки, но положение самих разделителей может не совпадать, это будет исправлять форматер. Текст внутри может быть многострочным, к нему применяются те же системы, что и ко всему файлу.

Также любой колонке можно дать имя и тип чтобы к ней можно было обращаться из расширений. Для этого нужно добавить к разделителю строк который форматирует верхнюю ячейку новую строку. В которой левый разделитель начальной ячейки будет `|` и через `-` будет указанны `[name]type`, оставшееся пространство до следующего разделителя следует заполнить, `-` число столбцов должно совпадать с разделителем строк выше, но разделители колонок не обязаны, это поправит форматер. При этом все разделители этой колонки должны состоять только из `|` ибо форматирование ячеек наследуется из первой. А сами ячейки теперь не требуют построчных разделителей ибо их данные занимаю 1 строку (пустые строки игнорируются). Но важно, хоть и рекомендуется использовать эту систему не для всех колонок в рамках одних и тех же строк, это возможно, но требует строгого соблюдения числа строк. Также можно создавать не именованные данные, в таком случае первый постоянный символ - в разделителе над первой ячейкой нужно заменить на 'd'

У таблицы есть параметр `size` в котором указан список размеров колонок (сумма элементов в нём должна быть равна 1, а их число должно совпадать с числом колонок)

После таблицы в кавычках идёт описание таблицы, тест внутри имеет обязательный отступ если он многострочный.

В примере ниже первая буква указывает на то как будет выравнен текст по вертикали, вторая по горизонтали. В ячейках на которые ссылается пояснение стоят их номера.
```
[table name]
{size = [50%, 20%, 30%]}
^:-------------|------v-----:|
| вл           | цц   | нп 1 |
|---------------------| нп 1 |
| цц 2                | нп 1 |
|-------------:^-------------|
| цп           | вц          |
|-d-----------:|-------------|<<
|-[name]type---|-------------|
| цп 3         | цц 5        |
|              |:------------|
| цп 4         | цл 6        |
v-------------:|             |
| нп           |             |
"
    Таблица с самым полным примером синтаксиса (в реальности таких не будет ибо это абсурдное нагромождение всех функций).

    С начало указанно имя таблицы, потом соотношение размеров через параметр.
    Ячейка 1 объединена по вертикали так как разделитель строк под ней состоит не из `-`, её текст это: "нп 1\nнп 1\nнп 1".
    Ячейка 2 объединена по горизонтали так как разделитель строк над ней не содержит 1 разделителя колонок.
    Ячейки 3 и 4 это 2 ячейки именованного набора. Их должно быть 2 так как сбоку от них 2 ячейки 5 и 6, несмотря на то что ячейка 6 объединена по вертикали с ячейкой ниже
"

[table name 2]
|---|---------------|
| Имена | этапы ихмерений |
|   |---|---|---|---|
|   | 1 | 2 | 3 | 4 |
|-d:|-d-|-d-|-d-|-d-|<<
|-[name]-|-[e1]int-|-[e2]int-|-[e3]int-|-[e4]int-|
| Anedew | 30 | 40 | 20 | 50 |
| Bob | 30 | 40 | 20 | 50 |
| Alex | 30 | 40 | 20 | 50 |
| Jon | 30 | 40 | 20 | 50 |"
    То как напишет таблицу человек, до обработки форматером
"

[table name 2 (result)]
|--------|---------------------------------------|
| Имена  | этапы измерений                       |
|        |---------|---------|---------|---------|
|        | 1       | 2       | 3       | 4       |
|-d-----:|-d-------|-d-------|-d-------|-d-------|<<
| Anedew | 30      | 40      | 20      | 50      |
| Bob    | 30      | 40      | 20      | 50      |
| Alex   | 30      | 40      | 20      | 50      |
| Jon    | 30      | 40      | 20      | 50      |"
    То что выдаст форматер
"
```

## Блоки кода
Блоки кода это основной способ расширения возможностей редактора.
В специальном файле можно прописать какая программа исполнит этот код и как извлечь её результат. Но в конце концов результатом всегда будет текст в формате этого редактора и набор ресурсов (по типу изображений) которые этот текст обязан вставить.

Блоки кода прописываются, как и в markdown, но вместо языка указывается название расширения в котором настроено исполнение этого кода.

Например, так будут реализованны диаграммы, авто расчёт математики и LaTeX.
